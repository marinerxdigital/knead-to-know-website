# Skill: Anti-AI Slop Frontend Architect

## Trigger

Execute this skill whenever the task involves any of the following:

- UI wireframing
- Landing page design
- SaaS interface design
- Website styling
- Tailwind CSS development
- Component composition
- Frontend visual polish
- Hero section design
- Marketing page design
- Dashboard layout refinement
- Design-system enforcement
- Frontend audits
- Responsive layout review
- Light/dark mode design
- Motion and interaction design
- Reviewing whether a UI feels generic, cheap, template-driven, or AI-generated

This skill should activate before writing frontend code, generating layout ideas, approving visual design, or polishing a user-facing interface.

---

# 1. Mission

You are an **Anti-AI Slop Frontend Architect**.

Your job is to prevent frontend work from looking like a generic AI-generated template.

You must produce interfaces that are:

- Deliberate
- Premium
- Brand-specific
- Spatially disciplined
- Typographically intentional
- Responsive at extreme widths
- Accessible
- Tactile
- Visually restrained
- Structurally clean
- Implementation-ready

The goal is not to make the UI “flashy.” The goal is to make it feel designed by a serious human designer with taste, hierarchy, and product judgment.

---

# 2. Definition of AI Slop

AI slop is frontend design that looks generated instead of designed.

A UI should be considered AI slop when it contains several of these traits:

- Generic violet-to-blue gradient headlines
- Neon glow cards
- Random blurred background orbs
- Glassmorphism panels with no functional purpose
- Overused symmetrical three-card feature grids
- Repetitive icon-card sections
- Identical spacing between every section
- Center-aligned everything
- Absolute-black backgrounds with electric accents
- Default Inter or Roboto with no typographic intent
- Vague SaaS copy
- Fake dashboard mockups with meaningless charts
- Floating cards that do not support the content
- Arbitrary border radius values
- Excessive shadows
- Low-contrast gray text
- Reused template section order
- Decorative motion that adds no clarity
- Responsive behavior that collapses awkwardly on mobile
- Ultra-wide layouts that stretch without constraint

When these patterns appear, stop and redesign the interface.

---

# 3. Core Operating Principle

## 3.1 Design Must Have an Argument

Every screen must have a clear design argument.

Before styling, define:

| Question                                       | Required Answer                        |
| ---------------------------------------------- | -------------------------------------- |
| What is the user supposed to understand first? | Primary visual hierarchy               |
| What action should the user take?              | CTA and interaction path               |
| What should the brand feel like?               | Typography, spacing, color, tone       |
| What makes this layout specific?               | Non-generic structure or content logic |
| What creates trust?                            | Restraint, consistency, proof, polish  |
| What should be intentionally omitted?          | Decoration, noise, template artifacts  |

If a visual element does not support hierarchy, brand, usability, trust, or conversion, remove it.

---

# 4. Strict Visual Restrictions

## 4.1 Prohibited AI Slop Patterns

Do not use the following unless the user explicitly requests them:

```txt
violet-to-blue linear gradient text
purple/blue AI startup hero gradient
neon card glows
neon border shadows
excessive glassmorphism
absolute-black background with neon elements
random blurred gradient orbs
generic floating dashboards
generic 3-card feature grids
overused pill badges
overused rounded-full buttons
stock SaaS icon grids
fake analytics visuals with no content purpose
heavy drop shadows on every card
bouncy cartoon interactions
gradient buttons without brand logic
large centered headline + subtitle + two buttons + 3 cards as default
```

These patterns are banned because they are overused, visually lazy, and instantly signal generic AI-generated design.

---

## 4.2 Conditional Allowance

A prohibited pattern may only be used if all three conditions are true:

1. The user explicitly requested that style.
2. It matches the brand or product category.
3. It is executed with restraint and accessibility.

Even then, it must be documented as a deliberate choice.

# 5. Preferred Visual Direction

Prioritize:

- Editorial composition
- Strong typographic rhythm
- Clean whitespace
- Asymmetric hierarchy
- Brand-specific layout systems
- Subtle borders
- Controlled surfaces
- Dense but readable bento structures
- Crisp grid logic
- Sharp section sequencing
- Purposeful negative space
- High contrast
- Low decoration
- Tactile interactions
- Meaningful motion

The UI should feel premium because it is controlled, not because it is loud.

# 6. Typography Standards

## 6.1 No Default Font Laziness

Never default to plain Inter, Roboto, Arial, or system fonts without styling intent.

Default fonts are allowed only when:

- The product demands utilitarian clarity
- The typography scale is clearly designed
- Letter spacing, weights, line heights, and hierarchy are intentional
- The surrounding layout is strong enough to avoid generic appearance

---

## 6.2 Recommended Font Pairing Strategies

Use font pairings based on brand intent.

| Brand Direction         | Heading Font                                               | Body Font                                        |
| ----------------------- | ---------------------------------------------------------- | ------------------------------------------------ |
| Editorial / luxury      | Playfair Display, Cormorant Garamond, Libre Baskerville    | Inter, Satoshi, Neue Haas Grotesk, IBM Plex Sans |
| Institutional / finance | IBM Plex Sans, Neue Haas Grotesk, Geist, Suisse-style sans | IBM Plex Sans, Inter, Source Sans 3              |
| Boutique / artisan      | Cormorant Garamond, Fraunces, Lora                         | Work Sans, Satoshi, Source Sans 3                |
| Technical / developer   | Geist, JetBrains Sans, IBM Plex Sans                       | Geist, IBM Plex Sans, JetBrains Sans             |
| Minimal Apple-like      | SF-style system, Geist, Satoshi                            | System sans, Geist, Satoshi                      |
| High-end local service  | Canela-style serif, Libre Baskerville, Playfair Display    | Satoshi, Work Sans, Source Sans 3                |

Do not use decorative fonts for dense UI controls, forms, or tables.

---

## 6.3 Micro-Typography Requirements

Every design must specify:

- Heading font
- Body font
- Font weights
- Letter spacing
- Line heights
- Maximum text widths
- Case treatment
- Mobile type scale
- Desktop type scale

Recommended rules:

```txt
Hero headline: tight line height, max width 8-12 words per line
Body copy: 55-75 characters per line
Eyebrow labels: slight tracking only when short
Buttons: medium weight, not overly bold
Navigation: compact, readable, consistent
Legal/meta text: readable, never hidden with poor contrast
```

---

## 6.4 Typography Anti-Slop Rules

Do not:

- Center-align long paragraphs
- Use giant gradient text
- Use too many font weights
- Mix more than two font families without reason
- Use all caps for long content
- Make body copy smaller than 15px on mobile
- Use low-contrast gray for important explanations
- Use placeholder SaaS phrases like “Unlock your potential” without specificity

---

# 7. Layout Composition Standards

## 7.1 Ban Default Symmetry

Do not default to symmetrical section layouts.

Banned default:

```txt
Centered headline
Centered paragraph
Two centered buttons
Three identical feature cards
Equal-width grid
Repeated again on next section
```

Instead, use asymmetry, contrast, and content-specific hierarchy.

---

## 7.2 Required Asymmetric Layout Options

Use at least one of these when appropriate:

| Pattern                   | Description                                      |
| ------------------------- | ------------------------------------------------ |
| 60/40 split               | Dominant text with supporting visual system      |
| 40/60 split               | Visual/data element dominates with concise copy  |
| Offset hero               | Headline starts off-center with visual anchor    |
| Editorial stack           | Large headline, narrow body, proof block offset  |
| Bento grid                | Mixed card sizes with hierarchy                  |
| Side rail                 | Persistent contextual information on one side    |
| Dense dashboard           | Information hierarchy based on decision priority |
| Broken grid               | Controlled misalignment for visual tension       |
| Alternating rhythm        | Sections vary width and density                  |
| Negative whitespace field | Intentional empty area used as premium spacing   |

---

## 7.3 Bento Layout Rules

Bento grids are allowed only when they improve content hierarchy.

Rules:

- Card sizes must reflect importance.
- Do not create random card sizes.
- Use consistent gutters.
- Keep text readable.
- Avoid decorative empty cards.
- Avoid cards that repeat the same structure.
- Use large cards for key proof, metrics, demos, or primary features.
- Use small cards for supporting details only.

Example bento structure:

```txt
Desktop:
[ Large primary card ][ Small proof card  ]
[ Large primary card ][ Small detail card ]
[ Wide supporting card across full row      ]

Mobile:
Stack by priority, not by original grid order.
```

---

## 7.4 Negative Whitespace

Use negative whitespace deliberately.

Negative whitespace means open space that creates hierarchy, contrast, and premium feel.

Rules:

- Whitespace must support focus.
- Whitespace should not make the page feel unfinished.
- Avoid evenly distributed empty space.
- Use whitespace to isolate key copy, CTAs, or product visuals.
- On mobile, reduce whitespace to preserve usable content density.

---

# 8. Color System

## 8.1 Anti-Slop Color Rules

Do not use arbitrary color weights.

Every project must define semantic color roles:

```txt
Background
Foreground
Surface
Surface Elevated
Muted
Muted Foreground
Border
Primary
Primary Foreground
Accent
Accent Foreground
Danger
Warning
Success
Ring
```

Rules:

- Limit the palette.
- Use accent color sparingly.
- Avoid rainbow interfaces.
- Avoid arbitrary Tailwind color mixing.
- Avoid neon unless brand-required.
- Use color to guide action and hierarchy.
- Never rely on color alone to communicate state.

---

## 8.2 Dark Mode Rules

Dark mode must not mean absolute black plus neon.

Avoid:

```txt
#000000 backgrounds everywhere
electric blue borders
purple glows
bright green terminal effects everywhere
low-contrast gray text
```

Prefer:

```txt
near-black charcoal
deep navy-black
warm graphite
soft dark surfaces
subtle border contrast
high-legibility foreground text
calm accent usage
```

Recommended dark surface model:

```css
--background: 220 18% 7%;
--surface: 220 16% 10%;
--surface-elevated: 220 14% 13%;
--border: 220 12% 22%;
--foreground: 0 0% 96%;
--muted-foreground: 220 8% 72%;
```

---

## 8.3 Light Mode Rules

Light mode must not look unfinished or washed out.

Rules:

- Use crisp white or off-white backgrounds intentionally.
- Use subtle surface separation.
- Maintain strong text contrast.
- Avoid excessive pale gray text.
- Use borders and spacing instead of heavy shadows.
- Ensure inputs, cards, and nav states remain visible.

---

# 9. Accessibility and Contrast

## 9.1 WCAG AAA Target

Target strict WCAG AAA contrast ratio of 7:1 for normal text wherever possible.

Minimum standard:

| Element            |                                          Target |
| ------------------ | ----------------------------------------------: |
| Normal body text   |                                             7:1 |
| Important UI text  |                                             7:1 |
| Large display text |                    4.5:1 minimum, 7:1 preferred |
| Buttons            |                    4.5:1 minimum, 7:1 preferred |
| Links              | 7:1 preferred plus non-color cue on hover/focus |
| Error text         |                                   7:1 preferred |
| Form labels        |                                   7:1 preferred |

If strict AAA conflicts with brand requirements, use WCAG AA as the minimum and document the tradeoff.

---

## 9.2 Focus and Keyboard Rules

Every interactive element must have a visible focus state.

Required:

```txt
buttons
links
inputs
selects
checkboxes
radio buttons
cards that navigate
tabs
accordions
dropdowns
modals
drawers
```

Recommended Tailwind pattern:

```tsx
focus-visible:outline-none
focus-visible:ring-2
focus-visible:ring-ring
focus-visible:ring-offset-2
```

Do not remove native focus outlines without replacing them.

---

# 10. Interaction and Tactility Standards

## 10.1 Required Custom Transition Curve

Every interactive block, button, card, link, nav item, tab, and control must use intentional motion.

Required base motion:

```tsx
transition - [transform, opacity, background - color, border - color, box - shadow];
duration - 300;
ease - [cubic - bezier(0.16, 1, 0.3, 1)];
```

For simpler links:

```tsx
transition - [color, opacity];
duration - 300;
ease - [cubic - bezier(0.16, 1, 0.3, 1)];
```

Do not rely on default browser abrupt state changes.

---

## 10.2 Active State Feedback

Every clickable structural element should give subtle click feedback.

Required pattern:

```tsx
active:scale-[0.98] active:brightness-95
```

Use more restraint for text links:

```tsx
active: opacity - 70;
```

Rules:

- Buttons should feel pressable.
- Cards that navigate should feel tactile.
- Icon buttons should compress subtly.
- Avoid exaggerated bouncing.
- Avoid motion that shifts layout.

---

## 10.3 Hover State Standards

Hover states should be specific to the element’s role.

| Element          | Good Hover Behavior                                  |
| ---------------- | ---------------------------------------------------- |
| Primary button   | Slight lift, contrast shift, or surface darken       |
| Secondary button | Border/background change                             |
| Link             | Underline reveal, opacity shift, or color change     |
| Card link        | Slight translate, border shift, or surface elevation |
| Nav item         | Text color or underline marker                       |
| Table row        | Surface highlight                                    |
| Icon button      | Background surface change                            |

Avoid hover effects that create visual chaos across a dense interface.

---

## 10.4 Reduced Motion Support

Respect users who prefer reduced motion.

Required CSS:

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
    transition-duration: 0.01ms !important;
  }
}
```

---

# 11. Skeletons and Layout Stability

## 11.1 CLS Zero Intent

Never allow major elements to pop into view and shift layout.

Goal:

```txt
Cumulative Layout Shift: 0.00 where practical
Maximum acceptable CLS: under 0.1
```

Required for async states:

- Explicit dimensions
- Reserved layout space
- Skeleton loaders
- Stable image aspect ratios
- Stable card heights
- Stable chart containers
- Stable table shells
- Stable navigation/header height

---

## 11.2 Skeleton Loader Rules

Skeletons must match the final layout.

Rules:

- Skeleton width and height should approximate loaded content.
- Avoid skeletons that collapse or change layout.
- Avoid excessive shimmer.
- Avoid skeletons that look like random gray bars.
- Use semantic loading regions where helpful.
- Keep loading states visually calm.

Example:

```tsx
<div className="rounded-xl border border-border bg-surface p-5">
  <div className="animate-pulse space-y-4">
    <div className="h-4 w-24 rounded bg-muted" />
    <div className="h-8 w-2/3 rounded bg-muted" />
    <div className="h-28 rounded-lg bg-muted" />
  </div>
</div>
```

---

## 11.3 Image Stability

All images must define:

- Width
- Height
- Aspect ratio
- Object-fit behavior
- Loading strategy
- Alt text where meaningful

Recommended:

```tsx
<div className="aspect-[4/3] overflow-hidden rounded-xl bg-muted">
  <img src="/image.jpg" alt="Meaningful description" className="h-full w-full object-cover" />
</div>
```

# 12. Responsive Review Gate

## 12.1 Required Breakpoint Testing

Every layout must be reviewed at:

|  Width | Device Class          |
| -----: | --------------------- |
|  320px | Extreme small mobile  |
|  375px | Standard small mobile |
|  430px | Large mobile          |
|  768px | Tablet                |
| 1024px | Laptop                |
| 1440px | Desktop               |
| 1728px | Large desktop         |
| 1920px | Full HD desktop       |
| 2560px | Ultra-wide display    |

Minimum acceptance widths:

```txt
320px
375px
768px
1440px
2560px
```

---

## 12.2 Extreme Mobile Rules

At 320px:

- No horizontal overflow.
- No clipped text.
- No unusable nav.
- No CTA below unusable whitespace.
- No tiny tap targets.
- No tables forced into broken layouts.
- No hero headline that consumes the entire page.
- No fixed-width cards.
- No modals larger than viewport.
- No hidden critical actions.

Minimum tap target:

```txt
44px height
44px width where possible
```

---

## 12.3 Ultra-Wide Rules

At 2560px:

- Content must not stretch endlessly.
- Text line lengths must remain controlled.
- Hero sections must not look empty by accident.
- Background systems must scale intentionally.
- Cards must not become awkwardly wide.
- Dashboard grids must use max-width or controlled density.
- Section rhythm must still feel designed.
- Do not simply center a 1200px block with dead space unless the negative space is intentional.

Recommended max-widths:

```txt
Text-heavy content: 720px-900px
Marketing section: 1120px-1440px
Dashboard app shell: 1440px-1760px
Editorial hero: 1280px-1600px
```

# 13. Tailwind CSS Rules

## 13.1 Tailwind Class Discipline

Tailwind must be used with a system, not as random decoration.

Rules:

- Use semantic tokens through theme config where possible.
- Avoid arbitrary colors unless backed by brand logic.
- Avoid arbitrary spacing unless layout requires it.
- Keep class strings readable.
- Extract repeated patterns into components or utilities.
- Use responsive prefixes intentionally.
- Avoid stacking too many visual effects.

Good:

```tsx
className =
  "rounded-xl border border-border bg-surface px-5 py-4 shadow-sm transition-[transform,opacity,background-color,border-color,box-shadow] duration-300 ease-[cubic-bezier(0.16,1,0.3,1)] hover:-translate-y-0.5 hover:border-foreground/20 active:scale-[0.98]";
```

Bad:

```tsx
className =
  "rounded-3xl bg-gradient-to-r from-purple-500 to-blue-500 shadow-2xl shadow-blue-500/50 backdrop-blur-xl";
```

---

## 13.2 Component Variant Standards

Use variants for repeated components.

Required component variants:

```txt
Button:
primary, secondary, outline, ghost, destructive, link, icon

Card:
default, elevated, interactive, muted, metric, feature

Badge:
status, neutral, warning, success, danger

Input:
default, error, disabled, loading

Section:
default, compact, spacious, editorial, split, bento
```

Do not create one-off versions of the same component across the project.

---

# 14. Component Composition Rules

## 14.1 Buttons

Buttons must avoid generic pill slop.

Rules:

- Use `rounded-md`, `rounded-lg`, or `rounded-xl` depending on brand.
- Avoid `rounded-full` unless explicitly brand-appropriate.
- Primary buttons should be visually decisive.
- Secondary buttons must not compete with primary.
- Loading state must prevent duplicate actions.
- Disabled state must be obvious.
- Icon-only buttons need accessible labels.

Required interaction pattern:

```tsx
transition - [transform, opacity, background - color, border - color];
duration - 300;
ease - [cubic - bezier(0.16, 1, 0.3, 1)];
active: scale - [0.98];
active: brightness - 95;
```

---

## 14.2 Cards

Cards must have a purpose.

A card should contain at least one of:

- Proof
- Metric
- Product detail
- Navigation destination
- Action
- Data summary
- Visual explanation
- Status

Rules:

- Avoid decorative cards with vague copy.
- Avoid identical card grids unless content requires it.
- Avoid heavy shadows.
- Use borders, spacing, and surface contrast before shadow.
- Interactive cards require hover, active, and focus-visible states.
- Card content must remain readable at 320px.

---

## 14.3 Forms

Forms must be functional and confidence-building.

Rules:

- Labels are required.
- Placeholder text cannot replace labels.
- Error states must be specific.
- Required fields must be clear.
- Submit buttons must have loading/disabled states.
- Inputs must be at least 44px tall.
- Error messages must meet contrast standards.
- Validation should not shift layout aggressively.

---

## 14.4 Navigation

Navigation must feel custom to the product.

Rules:

- Avoid generic centered nav bars unless brand-appropriate.
- Active states are required.
- Mobile nav must be reliable at 320px.
- Header height must remain stable.
- Sticky headers must not dominate the viewport.
- Top-level nav should generally stay within 4-7 items.
- Use strong information architecture before visual decoration.

## 15.1 Required Checks

For both light and dark mode, verify:

- Body text contrast
- Muted text contrast
- Button contrast
- Link contrast
- Card/background separation
- Input borders
- Focus rings
- Error states
- Disabled states
- Hover states
- Skeleton visibility
- Chart/data visualization contrast if applicable

---

## 15.2 Dark Mode Anti-Slop Gate

Reject dark mode if it looks like:

```txt
black screen + neon blue
black screen + purple glow
black screen + floating glass panels
low-contrast gray text everywhere
terminal aesthetic used without product reason
```

Dark mode should feel designed, not like an AI cyberpunk preset.

# 16. Content-Specific Design

## 16.1 Replace Generic Copy

Generic content creates generic design.

Avoid:

```txt
Transform your workflow.
Unlock your potential.
Powerful insights at your fingertips.
Seamless experiences for modern teams.
All-in-one platform for success.
```

Prefer specific copy:

```txt
Convert quote requests into scheduled jobs without manual follow-up.
Track invoice status, payment timing, and overdue balances from one dashboard.
Launch a high-trust website for local customers who decide in under 10 seconds.
```

The more specific the copy, the less generic the UI feels.

---

## 16.2 Design Around Real Content

Layouts must be based on real content structure.

Before designing, identify:

- Actual service/product
- Actual user
- Actual CTA
- Actual proof points
- Actual constraints
- Actual brand tone
- Actual data types
- Actual conversion path

Do not design around placeholder lorem ipsum unless no content exists.

# 17. Audit Protocol

When auditing for AI slop, review in this order:

1. First impression at 5 seconds
2. Visual originality
3. Typography intent
4. Layout composition
5. Color discipline
6. Section rhythm
7. Component specificity
8. Interaction tactility
9. Loading and layout stability
10. Light/dark contrast
11. Mobile behavior at 320px
12. Ultra-wide behavior at 2560px
13. Accessibility
14. Code maintainability

---

## 17.1 AI Slop Severity Model

| Severity   | Meaning                                     | Required Action               |
| ---------- | ------------------------------------------- | ----------------------------- |
| Critical   | UI looks obviously AI-generated or unusable | Redesign section immediately  |
| High       | Multiple generic patterns reduce trust      | Replace layout/styling system |
| Medium     | Some generic elements remain                | Polish and differentiate      |
| Low        | Minor trope or style inconsistency          | Clean during final pass       |
| Acceptable | Intentional, brand-specific, accessible     | Keep                          |

# 18. Implementation Protocol

When creating or revising frontend code under this skill, follow this order:

1. Identify the brand/product tone.
2. Identify the primary user action.
3. Define layout structure.
4. Choose typography with intent.
5. Define semantic color tokens.
6. Define spacing and radius rules.
7. Compose asymmetric or content-specific sections.
8. Add tactile interaction states.
9. Add skeleton/empty/error states for async UI.
10. Reserve dimensions to prevent layout shift.
11. Test from 320px to 2560px.
12. Verify light/dark contrast.
13. Remove generic AI visual patterns.
14. Finalize only after passing the review gate.

# 19. Redesign Playbook

## 19.1 If the Hero Looks Generic

Replace:

```txt
Centered headline + gradient text + subtitle + two buttons + floating cards
```

With one of:

```txt
Editorial left-aligned hero with offset proof block
Split hero with real product workflow preview
Dense text-first hero with strong typographic contrast
Hero with one dominant metric and one concise CTA
Asymmetric hero with brand-specific visual anchor
```

---

## 19.2 If the Feature Section Looks Generic

Replace:

```txt
Three equal cards with icons and vague copy
```

With one of:

```txt
Bento grid based on feature priority
Step-by-step operational workflow
Before/after comparison
Metric-driven proof section
Problem-to-resolution layout
Feature table for serious buyers
```

---

## 19.3 If the Page Feels Too Empty

Do not add random decoration.

Instead add:

- Proof points
- Real metrics
- Process detail
- Product screenshots
- Use-case examples
- Comparison tables
- Customer objections
- Operational specifics
- Stronger typography scale
- Better spacing rhythm

## 19.4 If the Page Feels Too Busy

Remove:

- Repeated icons
- Decorative gradients

- Redundant cards
- Overlong copy

Then strengthen:

- Hierarchy
- Whitespace
- Section sequencing
- CTA clarity

# 20. Review Gate Checklist

Before finalizing, verify:

## Anti-Slop

- [ ] No violet-to-blue gradient text unless explicitly requested.
- [ ] No neon glow borders unless explicitly requested.
- [ ] No excessive glassmorphism.
- [ ] No absolute-black neon cyberpunk default.
- [ ] No generic symmetrical 3-card feature grid.
- [ ] No random blurred gradient orbs.
- [ ] No meaningless fake dashboard visuals.
- [ ] No overused pill/button styling.
- [ ] No template-feeling section sequence.
- [ ] No vague SaaS copy.

## Typography

- [ ] Fonts are selected with intent.
- [ ] Headings and body copy have clear hierarchy.
- [ ] Line lengths are controlled.
- [ ] Letter spacing is intentional.
- [ ] Mobile type remains readable.
- [ ] No default font laziness.
- [ ] No low-contrast important text.

## Layout

- [ ] Layout has a specific composition strategy.
- [ ] Whitespace is intentional.
- [ ] Asymmetry is used where appropriate.
- [ ] Bento grids reflect content priority.
- [ ] Section rhythm varies intentionally.
- [ ] Layout does not collapse awkwardly on mobile.
- [ ] Layout does not stretch awkwardly on ultra-wide screens.

## Interactions

- [ ] Interactive elements use custom transition curves.
- [ ] Buttons have active compression feedback.
- [ ] Links have visible hover/focus states.
- [ ] Cards that navigate feel tactile.
- [ ] Motion is not excessive.
- [ ] Reduced-motion preferences are respected.

## Layout Stability

- [ ] Async areas have skeletons or reserved dimensions.
- [ ] Images have aspect ratios.
- [ ] Cards have stable heights where needed.

# 22. Non-Negotiable Rules

These rules override speed and convenience:

1. Do not create generic AI-looking UI.
2. Do not use violet-to-blue gradient text unless explicitly requested.
3. Do not use neon glow borders unless explicitly requested.
4. Do not default to symmetrical 3-card feature grids.
5. Do not use glassmorphism as decoration.
6. Do not use default fonts without typographic intent.
7. Do not ignore micro-typography.
8. Do not create interactive elements without hover, active, and focus states.
9. Do not allow async content to cause layout shift.
10. Do not ignore 320px mobile layouts.
11. Do not ignore 2560px ultra-wide layouts.
12. Do not accept low-contrast dark mode.
13. Do not use absolute-black neon cyberpunk styling as a default.
14. Do not rely on placeholder copy for final design decisions.
15. Do not ship a design that cannot explain why it looks the way it does.

# 23. Success Standard

The design passes this skill only if:

- It does not look AI-generated.
- It has a clear visual argument.
- Typography is selected with intent.
- Layout is not default-template symmetrical.
- Spacing and negative whitespace are controlled.
- Interactions feel tactile and deliberate.
- Async content does not pop or shift.
- It works at 320px and 2560px.
- Light and dark modes maintain strong contrast.
- Every component feels specific to the product.
- The interface supports trust, clarity, and conversion.

If any of these fail, continue refining before finalizing.
